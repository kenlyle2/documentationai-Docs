## Repositories

| Repo | Purpose |
|---|---|
| `kenlyle2/postglider` | Main Next.js app, edge functions, migrations |
| `kenlyle2/pg-media-vault` | WordPress plugin adapter + gallery-search edge function source |
| `kenlyle2/supawp-pg` | WordPress billing bridge plugin (fork of techcater/SupaWP) |

All three repos share the same backend project. The WP repos are canonical sources for their plugins; deployed edge functions always live in the main project.

## App Structure

```
app/
  (app)/              ← authenticated pages (dashboard, generate, gallery, etc.)
    generate/         ← Content Studio — XState machine drives all state
    gallery/          ← Media Vault
    packs/            ← Week Packs
    settings/         ← Profile, competitors, WP token
    billing/          ← Credits, subscription link
    admin/            ← Admin panel (is_admin = true only)
  api/                ← Next.js Route Handlers
    auth/             ← claim, callback
    admin/            ← tag-backlog, bulk operations
    intelligence/     ← competitor signals, day/format/hashtag signals
    places/           ← Google Places lookup
  start/              ← /start onboarding (public, anonymous session)
lib/
  actions/            ← Server Actions (galleryActions, etc.)
  machines/           ← XState v5 postgliderMachine (generate page state)
  db/                 ← client.ts, server.ts, middleware.ts
```

## XState in the Generate Page

The `/generate` page is driven by a formal XState v5 state machine at `@/lib/machines/postgliderMachine.ts`. All logical switches — loading states, modal visibility, active stage — are driven by `state.matches('state_name')`, not local booleans.

State flow: `initializing → idle → generating_text → tinder_review → viewing_idea → planning_image → generating_image → selecting_variant → error`

Realtime listeners for image updates trigger XState events (`VARIANTS_ARRIVED`) rather than setting local booleans.

## Intelligence Pipeline

| Signal | Source | Used by |
|---|---|---|
| Competitor signals | Competitor extractor → `competitor_signals` table | JIT nudges on Generate page |
| Day signal | `day_signal` edge function (computed) | Genie, content balance |
| Format signal | `format_signal` edge function (computed) | Genie suggestions |
| Hashtag signal | `hashtag` edge function → `competitor_signals` | Genie hashtag recommendations |

## Pilot Mode Drip Gate

`pilot_eligible` on the `profiles` table is set to `true` by a daily pg_cron job (`pilot-mode-drip`, runs midnight UTC). Threshold is configurable in `admin_settings` under `feature_drip.pilot_mode_days` (default: 7 days).

The column has `REVOKE UPDATE` from the `authenticated` role — users cannot self-elevate via the browser SDK. A backfill migration runs on deployment to immediately grant eligibility to accounts that already qualify.
